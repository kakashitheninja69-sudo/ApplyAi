Dead Import

resumeStore imports firestore.ts

resumeStore.ts imports ResumeListItem from @/lib/firestore, but the active persistence layer is now localResumes.ts. The Firestore import is unused and will fail if Firestore isn't enabled. Change to import from @/lib/localResumes.

Dead Import

AuthModal still imports firestore.ts

AuthModal.tsx imports loadUserResumes and loadResume from @/lib/firestore. These were replaced with localResumes.ts equivalents but the import was never updated, leaving a dead Firestore reference.

Dead Import

BuilderPage imports firestore.ts

BuilderPage.tsx imports saveResume and saveVersion from @/lib/firestore. The save calls were migrated to localResumes.ts but the import line still references Firestore.

Naming

Builder step file names are off-by-one

BuilderPage imports Step2Contact as "Step 1", Step3Work as "Step 2", etc. The file names don't match the logical step numbers. Rename files to match their position: Step1Contact, Step2Work, etc. (or accept the legacy offset).

Unused File

Step1Template.tsx is not used

src/components/builder/steps/Step1Template.tsx still exists in the file system but was removed from STEP_COMPONENTS in BuilderPage — the template is now chosen in the OnboardingPage. Safe to delete.

Bundle Size

Main chunk is 1.6 MB unminified

All 25 resume template components are statically imported in TemplateThumbnail.tsx, loading them all up-front. Consider lazy-loading templates with React.lazy() or dynamic imports — most users only ever view 1–2 templates.

Architecture

firestore.ts is dormant dead code

src/lib/firestore.ts is a full Firestore implementation that isn't wired up to any active feature. Either remove it to reduce confusion, or add a comment noting it's the future cloud upgrade path.

State

savedResumes in store is never populated

useResumeStore has a savedResumes: ResumeListItem[] field that was originally populated by a Firestore call. After migrating to localStorage, nothing calls setSavedResumes. Either remove the field or wire it up in DashboardPage.

UX

Dashboard requires no auth gate

DashboardPage previously redirected to / if currentUser was null. The migration removed the redirect, so any user (logged-out) can now visit /dashboard and see their local resumes. This is fine UX-wise, but the header's avatar block still renders conditionally, so it's inconsistent.
